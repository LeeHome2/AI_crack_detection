## 1. 설계 원칙 (핸드오프에서 확정된 제약 반영)

- **균열(ConcreteCrack) 단일 클래스**만 — 철근노출·누수·현장위험 범위 외
- **점수는 항상 Rule Engine(코드)이 매김** — YOLO·RAG·LLM은 "재료"만 제공
- **RAG 출력은 점수가 아니라 근거 문장**
- **mm 정밀 측정 안 함** — 픽셀 상대값으로 처리
- **1인·로컬 실행 OK** — 화려한 대시보드 지양, 핵심(분석결과·지표·근거)만
- 부사장 핵심: "AI는 판단 근거를 돕고 보고서 초안을 만드는 것"

## 2. 기술 스택

| 계층 | 선택 | 이유 |
|---|---|---|
| UI/앱 | **Streamlit** | 1인 MVP 최속. 사진 업로드·박스 이미지·지표·근거표를 단일 파일로 |
| 탐지 | **YOLOv8s (타일 학습본)** + 타일 슬라이스 추론 | 고해상도 균열선 보존 (검증됨: mAP50 0.118→0.179) |
| 형태 분석 | **OpenCV** (adaptive threshold + skeletonize) | 박스 내부 균열선/길이/폭 정량화 |
| 위험도 | **Rule Engine (순수 파이썬)** | 신뢰도·개수 기반 점수→등급. 설명가능·재현가능 |
| 지식검색 | **RAG**: ChromaDB(벡터DB) + 한국어 임베딩 | 안전기준 문서에서 근거 문장 검색 |
| 임베딩 | **BGE-m3 / multilingual-e5** (오픈소스, 한국어 지원) | 법령·지침이 한국어라 한국어 임베딩 필수. API비용 0 |
| 보고서 | **Anthropic Claude API** (claude-sonnet-5) | 비전문가용 서술 + 조치 권고 초안 생성 |

> 임베딩을 오픈소스로 두면 RAG는 API 비용·의존성 없이 로컬 동작. Claude API는 최종 보고서 문장 생성에만 사용 → 호출 최소화·비용 절감.

## 3. 시스템 아키텍처 (전체 파이프라인)

```
[1] 사진 업로드 (Streamlit)
        │  고해상도 1장
        ▼
[2] Vision AI — 균열 탐지 (detector.py)
        │  YOLOv8s 타일 학습본 + 타일 슬라이스 추론(overlap 20%) + NMS 병합
        │  출력: boxes[], confidence[]
        ▼
[3] Feature 추출 (features.py)
        │  각 박스 내부: 이진화→스켈레톤화→중심선
        │  출력: {crack_count, max_length_ratio, avg_width_px, ...}
        ▼
[4] RAG 기준 검색 (rag.py)          [5] Rule 위험도 산정 (rules.py)
        │  feature/키워드로 안전기준       │  신뢰도·개수 기반 점수→등급
        │  문서에서 근거 문장 top-k        │  (RAG 긴급기준 매칭 시 가점)
        │  출력: 근거 문장[] + 출처        │  출력: score, grade, 기여내역
        └──────────────┬──────────────────┘
                       ▼
[6] LLM 보고서 생성 (report.py)
        │  Claude API: feature+등급+근거를 비전문가용 서술로
        │  출력: 요약 / 위험도 설명 / 조치 권고 / 전문점검 권고
        ▼
[7] 결과 화면 (Streamlit)
           분석 이미지 · 위험도 등급 · 지표(F1/mAP) · RAG 근거 · 보고서 초안 · PDF 다운로드
```

핵심: **[4] RAG와 [5] Rule은 병렬**. Rule이 최종 점수를 매기고, RAG는 근거 문장을 대며, 필요 시 Rule의 "긴급기준 매칭" 항목에 신호만 전달.

## 5. 단계별 입출력 스키마 (schemas.py)

```python
# [2] 탐지 결과
Detection      = {"box": [x1,y1,x2,y2], "conf": float}
DetectResult   = {"detections": List[Detection], "image_size": [W,H]}

# [3] 형태 특징 (픽셀 상대값 기준 — mm 절대측정 안 함)
CrackFeatures  = {
    "crack_count": int,          # 균열 개수 (병합 후)
    "max_length_ratio": float,   # 최장 균열 길이 / 이미지 대각선
    "avg_width_px": float,       # 평균 폭(픽셀)
    "max_confidence": float,     # 최고 탐지 신뢰도
}

# [5] Rule 결과
RiskResult     = {
    "score": int,                # 0~100+
    "grade": str,                # 정상 / 주의 / 위험 / 긴급
    "contributions": List[dict], # 각 룰의 조건·가점 내역 (설명가능성)
}

# [4] RAG 결과
Evidence       = {"text": str, "source": str, "score": float}
RagResult      = {"evidences": List[Evidence]}

# [6] 보고서
Report         = {"summary": str, "risk_explain": str,
                  "actions": str, "inspection_advice": str}
```

## 6. Rule Engine 설계 (rules.py)

기획안 원안 기반, "현실성 있게" 조정 가능(부사장 승인). **균열만 구현.**

| Rule | 조건 | 점수 |
|---|---|---|
| 균열 탐지 | 최고 신뢰도 ≥ 0.80 | +30 |
| 균열 다수 | 균열 개수 ≥ 3 | +15 |
| 폭 상대값(선택) | avg_width_px 상위 구간 | +10 |
| RAG 긴급기준 매칭 | 정밀점검/긴급보수 기준 검색됨 | +20 |

**등급:** 0–39 정상 / 40–69 주의 / 70–89 위험 / 90+ 긴급
모든 가점은 `contributions`에 근거를 남겨 **왜 이 등급인지** 화면에 표시(설명가능성).

> ※ 철근노출(+30)·누수(+10)는 범위 외라 룰에서 제외. 표에는 "확장 예정"으로만 표기.

## 7. RAG 지식베이스 설계 (knowledge/)

**문서 소스 (한국어 공개 자료):**
- 국가법령정보센터: 시설물안전법 시행령/시행규칙
- 국토안전관리원: 안전점검·정밀안전진단 지침
- 콘크리트 허용균열폭 기준 (0.3mm = 표면처리/충진 분기)

**파이프라인:**
1. `build_index.py`: 원문 → 문단 청킹(300~500자, overlap) → 한국어 임베딩 → ChromaDB 저장 (1회)
2. 런타임 `rag.py`: feature/등급에서 쿼리 생성(예: "콘크리트 균열 폭 0.3mm 이상 조치") → top-k 유사 청크 → 근거 문장 반환
3. 결과 화면에 **근거 문장 + 출처(문서명·조항)** 표시 → 신뢰성 확보

핵심 원칙: **RAG는 점수를 매기지 않는다.** 근거 문장만 제공하고, 점수는 Rule이 담당.

---

## 8. 보고서 생성 설계 (report.py)

- 모델: **claude-sonnet-5** (Anthropic API)
- 입력: CrackFeatures + RiskResult + RagResult(근거)
- 프롬프트 원칙:
  - 비전문가(건물주·일반인) 대상 → 쉬운 서술
  - **근거는 RAG 문장 인용**, 점수는 Rule 결과를 그대로 전달(LLM이 점수 재산정 금지)
  - 과장·단정 금지, 마지막에 **"전문가 정밀점검 권고"** 고정 포함
- 출력 4블록: 요약 / 위험도 설명 / 권고 조치 / 전문점검 권고
- 비용·속도: 보고서 1건당 API 1회 호출. 임베딩은 로컬이라 API 비용은 보고서 생성에만 발생.

## 9. 화면 구성 (app.py, Streamlit)

단일 페이지, 세로 흐름:
1. 상단: 사진 업로드 버튼
2. 분석 결과: 원본 + 탐지 박스 오버레이 이미지
3. 위험도 카드: 등급(색상) + 점수 + Rule 기여 내역
4. 정량 지표: 모델 성능(mAP50/F1) + 이번 이미지 feature 요약
5. RAG 근거: 검색된 안전기준 문장 + 출처
6. 보고서 초안: Claude 생성 텍스트 + **PDF 다운로드**

> 평가 기준의 "분석결과 + F1 등 지표 + RAG 활용도 표시"를 한 화면에 충족.

## 10. 데이터 플로우 (요청→응답, 1회)

```
사용자 사진 1장
  → detector: 타일 분할 추론 → 박스 N개 + 신뢰도
  → features: 박스별 스켈레톤 → count/length/width (픽셀 상대)
  → (병렬) rules: score/grade   |   rag: 근거 문장 top-k
  → report: Claude API → 4블록 텍스트
  → Streamlit: 이미지+등급+지표+근거+보고서 렌더 → PDF 저장
```

전 과정 로컬 처리(임베딩·YOLO·OpenCV·Rule), **외부 호출은 보고서 생성 1회뿐** → 오프라인 시연 시 보고서만 목업으로 대체 가능.

## 11. 구현 우선순위 (핸드오프 일정 반영)

1. `detector.py` — 이미 predict_tiled.py 있음 → 함수화 (**거의 완료**)
2. `features.py` — OpenCV 스켈레톤 (다음 최우선)
3. `rules.py` — Rule Engine (독립적, 빠름)
4. `knowledge/build_index.py` + `rag.py` — 문서 10개+ 수집 후
5. `report.py` — Claude API 연동
6. `app.py` — Streamlit로 묶기

MVP 최소선: 1→2→3→6 (탐지→특징→위험도→화면). RAG/보고서는 그 위에 얹음.

## 12. 멘토링 논의 포인트

- 아키텍처 전반(특히 RAG=근거 / Rule=점수 분리)이 기획 의도와 맞는지
- Rule 임계값(신뢰도 0.80, 개수 3)이 현실적인지 → 정량 자료 공유 가능한지
- 보고서 초안의 기대 수준·형식(비전문가용 깊이)
- 대상 시설물 유형(건물 외벽 콘크리트 균열) 적절성
