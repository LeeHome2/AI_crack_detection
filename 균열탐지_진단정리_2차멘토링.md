# 균열탐지 YOLO 1차 학습 진단 정리

> 2차 멘토링(7/10 18:00) 대비 · 실제 데이터/학습결과 재검증본
> 결론: **타일 분할은 여전히 정답. 단, 근거를 "박스가 전체를 덮어서"에서 "다운스케일로 가는 균열이 소실돼서"로 교정**

## 1. 검증한 것 (실제 데이터 기준)

AI HUB 162번 라벨 JSON을 직접 스캔·집계하고, 1차 학습 결과(`train_v1`)와 검증 예측 이미지를 확인함.

- 라벨 JSON 총 **161,872개**
- 무작위 300개 표본으로 박스 통계 집계
- `runs/detect/val`의 예측 이미지로 실제 탐지 실패 양상 확인

## 2. 핸드오프 진단의 오류 교정

### 기존 진단 (7/8 핸드오프)
> "바운딩박스가 이미지 대부분을 덮음. 예시 `[1195,90,265,1350]` → 세로 94%.
> YOLO가 '균열 = 이미지 전체'로 학습 → 위치 학습 실패."

### 실제 데이터 (교정)

| 항목 | 실측값 (표본 300장) |
|---|---|
| 이미지당 균열 박스 개수 | 평균 **3.0개** (중앙 3, 최대 10) |
| 박스 면적 / 이미지 면적 | 중앙 **1.7%**, 평균 3.3% |
| 박스 종횡비(긴변/짧은변) | 중앙 3.1, 평균 3.8 |
| 이미지 50% 이상 덮는 박스 | **319개 중 1개 (0.3%)** |

**→ 라벨은 균열을 따라 "적당한 크기 박스 여러 개"로 잘 쪼개져 있음.**
`[1195,90,265,1350]`(세로 94%) 같은 큰 박스는 **희귀 예외**였고, 이걸 전체 경향으로 오해했음.
(검증 라벨 시각화 `val_batch0_labels.jpg`에서도 대부분 균열선을 따라간 중소형 박스로 확인됨.)

---

## 3. 그럼 mAP50이 왜 0.119인가 — 진짜 원인

### 학습 결과 (train_v1, YOLOv8s, 50 epochs)
```
mAP50 0.118 / mAP50-95 0.039 / Precision 0.194 / Recall 0.227
box_loss 2.44(1ep) → 1.99(50ep)   ← 거의 안 줄어듦 (학습 정체)
```

### 검증 예측 확인
`val_batch0_pred.jpg`에서 모델이 **박스를 거의 하나도 예측하지 못함**
(신뢰도가 임계값 아래로 붕괴 → Recall 0.22).

### 근본 원인: **해상도 다운스케일**
```
원본 2560×1440  →  학습 입력 640×640  (약 4배 축소)
```
- 균열은 폭이 수 픽셀인 **가늘고 긴 선형 객체**
- 4배 축소하면 균열선이 1px 이하로 뭉개져 **신호 자체가 소실**
- 모델이 배경과 균열을 구분할 특징을 못 잡음 → box_loss 정체, 예측 없음

**즉, 문제는 "박스 모양"이 아니라 "입력 해상도"였다.**

---

## 4. 해결책 — 타일 분할 (결론 동일, 근거 교정)

```
원본 2560×1440 → 640×640 타일로 분할 후 학습 (overlap 20% 권장)
```
- 각 타일은 원본 해상도를 유지 → **가는 균열선이 살아있음**
- 타일 안에서 균열이 적절한 크기 객체가 됨
- 데이터 수도 증가 (3,900장 → 수만 타일)
- 참고: SAHI(Slicing Aided Hyper Inference)로 학습·추론 모두 적용 가능

### 실측 결과 — before / after (동일 yolov8s 백본, 공정 비교)

| 지표 | 1차 (통짜 640) | 타일 subset(300장) | **타일 full(3,120장)** |
|---|---|---|---|
| mAP50 | 0.118 | 0.143 | **0.179** (+52%) |
| mAP50-95 | 0.039 | 0.048 | **0.062** (+59%) |
| Precision | 0.194 | 0.255 | **0.285** (+47%) |
| Recall | 0.227 | 0.205 | **0.242** (+7%) |

- 데이터를 늘릴수록 **모든 지표가 단조 상승** → 타일링 방향이 옳다는 실증
- full 학습: 69에폭 조기종료(best epoch 49), 약 4시간 (RTX 3070 Ti)

### mAP50 절대값이 낮은 이유 (지표 특성)
- 균열은 가는 대각선 → **axis-aligned 박스로 감싸면 정답·예측이 조금만 어긋나도 IoU<0.5** → mAP50에서 오답 처리
- 즉 **시각적으론 잘 잡아도 mAP는 잘 안 오르는** 구조. 여기에 AI HUB 라벨의 누락(보이는 균열 다수 미라벨)도 겹침
- → 이 프로젝트 목적(판단 근거·보고서)에 맞게 **"균열 유무 탐지(이미지 단위 recall)" + 시각적 탐지 품질**로 평가하는 게 타당

## 5. "균열을 선으로 추출" — 방법 검토

> 라벨에 **폴리곤/세그멘테이션이 없음** (JSON 키에 `segmentation` 부재, bbox만 존재).
> → 이 데이터로 YOLO-seg·U-Net 등 **세그멘테이션 지도학습은 불가**.

선을 얻는 현실적 경로:

| 방법 | 내용 | 평가 |
|---|---|---|
| **A. bbox 탐지 → OpenCV 스켈레톤** | 박스 안을 이진화→스켈레톤화해 중심선 추출 (핸드오프 feature 단계 그대로) | **추천.** 라벨 그대로 사용, 선·길이·폭 산출 가능 |
| B. 약지도 seg 학습 | GT 박스 내부를 CV로 마스크 자동생성 → 그 마스크로 YOLOv8-seg 학습 | 강력하나 마스크 생성·검수에 시간 소요 |
| C. 공개 seg 사전학습 모델 | 보유한 `YOLOv8-crack-seg` 활용 | 한국 건물 균열 도메인 불일치 → "before" 사례로만 |

## 6. 2차 멘토링 발표 스토리 (교정본)

1. 공개 사전학습 모델 2종 테스트 → 도메인 불일치 / 오탐 확인 (before)
2. AI HUB 건물 균열 데이터로 직접 학습 (1차: mAP50 0.118)
3. **진단:** 라벨은 정상(균열선 따라 중소형 박스). 진짜 병목은 **2560→640 다운스케일로 가는 균열선 소실** → 예측 붕괴
4. **해결·실증:** 640 타일 분할 + 전체 데이터 재학습 → mAP50 0.118→**0.179**(+52%), 전 지표 상승
5. mAP 절대값이 낮은 건 **가는 대각선+axis 박스의 IoU 한계** → "균열 유무 탐지 + 시각 품질"로 평가
6. 다음 단계: 박스 내부 OpenCV 스켈레톤으로 균열선/길이/폭 정량화 → Rule → RAG → 보고서

### 멘토링 질문 (유지)
- 균열 폭별 판정 기준 등 정량 자료 공유 가능한지 (RAG용)
- 보고서 초안의 기대 수준·형식 (비전문가용 깊이)
- 대상 시설물 유형(건물 외벽 콘크리트 균열) 적절성

### 참고 (오늘 발표 범위)
- 타일 재학습 완료 → **before/after 지표 확보** (mAP50 0.118→0.179)
- 내 사진 탐지(after)는 `predict_tiled.py`(타일 슬라이스 추론)로 생성 → 실내 벽 세로 균열을 선따라 추적, 가구 오탐 없음
- 다음: OpenCV 스켈레톤으로 균열선/길이/폭 정량화 → Rule/RAG/보고서
