# 재학습(v2) 가이드 — 3070 Ti에서 돌리기

## 1. 현재(v1) 모델 진단 요약

| 항목 | 값 |
|---|---|
| 백본 | YOLOv8s (11.14M), 단일 클래스 `crack` |
| 데이터 | AI Hub 162 "건물 균열 탐지 드론" (ConcreteCrack), 원본 ~3,120장 → 640 타일 |
| 학습 | 80ep 계획 → 69ep 조기종료(patience 20), best @ 61ep |
| 최종 | mAP50 **0.179** · mAP50-95 0.062 · precision **0.285** · recall **0.241** |

**핵심 진단:** mAP50이 40에폭(0.173)부터 68에폭(0.180)까지 거의 평평 → **이미 수렴한 천장.**
epoch만 더 돌려도 안 오른다. 낮은 원인은 (1) 드론↔폰 **도메인 불일치**, (2) precision 0.285
= 오탐 71%(타일 눈금·줄눈), (3) 얇은 균열 + bbox의 태생적 IoU 한계.

## 2. 데이터 판단 — "더 받아야 하나?"

같은 드론 데이터를 **더 받는 건 효용 낮음**(그 분포엔 이미 수렴). 효과 큰 순서:

1. **Hard negative 추가 (최우선, 라벨링 부담 0).** 균열이 *아닌* 배경 사진 —
   타일 바닥 줄눈·패널 경계·벽 몰딩·그림자·전선 자국 등 —을 라벨 없이 학습에 넣으면
   precision이 오르고 오탐이 모델 단에서 줄어든다. 방금 넣은 postprocess 필터와 **이중 방어**.
   목표: 300~800장(현장/집에서 폰으로 쉽게 수집).
2. **폰 근접 실사진 소량 라벨링 (도메인 갭 해소).** 실제 서비스 분포와 같은 폰 근접 사진
   100~300장을 라벨링(roboflow/labelImg)해 학습에 추가. 가성비 최고의 성능 개선.
3. (선택) 다른 크랙 데이터셋으로 다양성 보강 — AI Hub 타 균열셋, SDNET2018, Crack500 등.
   단, 1·2를 먼저 하고 효과 측정 후에.

## 3. 실행 순서

```bat
:: (1) hard negative 사진을 한 폴더에 모아둠 (예: D:\crack_detection\hard_neg)
::     - 타일 바닥, 줄눈, 패널 경계, 그림자 등 '균열 아님' 사진
:: (2) train_crack_tiled_v2.py 상단 HARD_NEG_DIR 를 그 경로로 설정
::     (스크립트가 라벨 없이 train/images 로 복사 → 배경으로 학습)
:: (3) 실행
(venv) D:\crack_detection> python train_crack_tiled_v2.py
```

폰 실사진을 추가할 때는 라벨링 후 `dataset_tiled_full\images\train` 및 `labels\train` 에
같은 파일명으로 넣고(빈 라벨=배경), 필요시 `data_tiled.yaml` 경로만 확인.

## 4. v2에서 바꾼 학습 설정 (근거)

| 파라미터 | v1 → v2 | 이유 |
|---|---|---|
| hard negative | 없음 → 추가 | precision↑, 타일/줄눈 오탐↓ |
| mosaic | 1.0 → 0.5 | 과한 조각모자이크가 얇은 균열 위치 흐림 |
| close_mosaic | 10 → 20 | 막판 20ep는 mosaic 꺼 실분포로 마무리 |
| scale | (기본) → 0.5 | 근접/원거리 촬영거리 다양성 |
| cos_lr | False → True | 후반 수렴 안정 |
| epochs/patience | 80/20 → 100/30 | 여유(수렴은 데이터가 좌우) |
| 백본 | 8s 유지 | v1과 **공정 비교** 먼저. 이후 8m(batch 8) 시도 |

## 5. 성공 판정 기준

숫자보다 **실사용 체감**이 우선. 확인 포인트:

- **Precision 상승**(0.285 → ?) = hard negative가 오탐을 줄였다는 신호(가장 중요).
- 폰 실사진 몇 장으로 before/after: 벽 한 줄이 1박스로, 바닥 타일이 안 잡히는지.
- mAP50은 bbox 태생적 한계로 크게 안 오를 수 있음 → precision/실사진 결과로 판단.

## 6. 더 큰 그림 (다음 라운드 후보)

얇은 선형 균열은 **segmentation(YOLOv8-seg)** 이 detection보다 근본적으로 잘 맞음
(픽셀 마스크 → 길이·폭 정확, bbox-IoU 페널티 없음). 중간점검(7/27) 이후 시간 여유가 있으면
seg 전환을 검토. 지금 일정에선 detection + postprocess가 실용적.
